## Key Observations from This Run

### O1 — `core_roles.json` remains empty despite the 5000-call run

`core_roles.json` is 2 bytes (contains `{}`). Its modification date is 2026-04-11 — it was not touched by this run at all. Meanwhile `not_relevant_core_roles.json` was updated with 665 KB of role data. This means:
- The roles were discovered and classified correctly (they appear in `not_relevant_core_roles.json`)
- The relevancy propagation still did NOT mark them as relevant
- This confirms that the `relevancy.py` kernel reload trap was at play: the fix was in the file but the kernel was using old code

### O2 — The preflight step alone consumed ~3,700 of 5,000 network calls

Step 2.4.3 alone used 3,247 calls. Combined with steps 2.4.1 and 2.4.2, roughly 3,700 of the 5,000 budget was spent before the main expansion (Step 6) even started. This means Step 6 had only ~1,300 calls available.

**Design implication:** The preflight class resolution walk is extremely expensive. 19,918 active classes required 3,247 network calls for the upward P279 branch discovery. A redesign must find a way to make this incremental (only process classes not yet resolved, not all 19,918 on every run).

### O3 — 19,918 active classes is the scale we are operating at

"Active classes" are the class QIDs referenced in P31/P279 claims of the 36,890+ discovered entities. With ~20k active classes, the graph is substantially larger than the "thousands of events" assumption the original architecture was built for.

### O4 — 2,860 active classes connected to core classes

Of 19,918 active classes, 2,860 were successfully connected to a core class via P279 upward walk. 13,511 entries in `class_resolution_map.csv`. This is the class hierarchy that relevancy propagation depends on.

### O5 — Step 2.4.3 is the single biggest bottleneck in cache-miss mode

~18 minutes for one preflight step (when class resolution is not already cached). In cache-hit mode (second run), the entire core subclass discovery is cache-only (5,634 pages served from cache, 0 network). But the **upward branch discovery** still required 3,201 network calls even with a warmed cache, because the P279 superclass pages for each of the 19,918 active classes may not be in cache.

**Design implication:** The redesign must make class resolution incremental: if a class was already resolved in a previous run, do not re-resolve it. EventHandler-based class resolution that tracks which classes it has already processed would reduce this to O(new_classes_since_last_run).

### O6 — The 4.5× active class explosion is caused by hydration ordering

Step 2.4 (before hydration) saw **4,446** active classes. Step 2.4.3 (after hydration) saw **19,918** — 4.5× more. The 2,961 entities fetched during step 2.4.2 introduced ~15,472 new class QIDs. This is the structural reason the two-pass preflight design exists. The problem is that both passes re-walk all active classes from scratch. Under incremental handler-driven class resolution, each entity discovery would enqueue only its *new* class QIDs.

### O7 — Step 2.4.2 (hydration) consumed 0 network calls

40,879 total hydration candidates; 2,917 were absent from the entity store and fetched from cache; 44 inactive-guarded entities activated. All 2,961 hydrations were cache hits — 0 network queries. This shows the cache is effective, but the design still required the two-pass preflight because class discovery followed hydration.

### O8 — Step 6 (main graph expansion) received only ~155 network calls

After all preflight steps, 1,315 calls remained. Step 6.5 started with `max_queries_per_run: 1160`, meaning Step 6 used approximately 155 calls. The core purpose of the notebook — expanding the graph from seeds — was budget-starved. 95%+ of the budget went to preflight and integrity.

### O9 — Step 6.5 (node integrity pass) ran for 63 minutes and processed only 35% of eligible seeds

1,525 eligible unexpanded QIDs. Only 530 expanded (35%) in 3,809 seconds. 1,547 total network queries (242 discovery + 1,305 expansion). Stop reason: `seed_complete` — the pass finished the last in-progress seed when budget neared exhaustion. 995 seeds remain unexpanded. The step also discovered **149 new QIDs** and repaired 1,709 discovery references — evidence that the integrity pass was doing real work (because entity references existed in triples without their source entity being fetched).

**Design implication:** The integrity-by-construction approach in the redesign (rule I1: every QID object in a stored triple must have entity data) eliminates this 63-minute post-hoc repair pass entirely.

### O10 — This run added essentially no new graph coverage

With Step 6 at ~155 calls and Step 6.5 budget-starved to 530 seeds, the run extended only marginally beyond the previous state. The large projection files (`core_organizations.json` at 244 MB) predate this run or were rebuilt from events, not freshly expanded. The `core_persons.json` was not updated at all (still from 2026-04-24). The run's primary effect was the Step 2.4.3 class resolution cache warm-up.
